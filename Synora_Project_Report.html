Abstract

The global events industry — spanning weddings, corporate conferences, cultural celebrations, product launches, and social gatherings — continues to experience rapid growth in both scale and complexity. Despite this growth, the technology supporting event coordination remains fragmented. Event organizers today rely on a disparate combination of messaging applications, spreadsheet tools, email clients, and phone calls to coordinate vendors, track budgets, and manage tasks — creating operational silos, communication gaps, and financial inefficiencies. No unified platform currently addresses the complete event management lifecycle from a mobile-first, intelligent perspective.

This report presents EVORA — an AI-Based Intelligent Event & Vendor Management System — developed as a cross-platform mobile application using the Flutter framework and Dart programming language. EVORA implements a Provider-based MVVM (Model–View–ViewModel) architecture for reactive, scalable state management and delivers a unified platform for three user roles: Administrators, Event Organizers, and Vendors. The system integrates a comprehensive suite of features including AI-assisted budget planning, conversational AI guidance, a structured vendor marketplace, in-app messaging, event task management, and an administrative vendor approval workflow.

Key innovations include: (i) the Nanban AI Chatbot — an intent-based conversational assistant providing real-time budget allocation advice and vendor recommendations; (ii) the Budget Recommendation Engine — a rule-based financial intelligence system that allocates budget percentages by expense category based on event type (Wedding, Birthday, Corporate) and generates smart over-budget insight alerts; (iii) a 13-category Vendor Marketplace supporting discovery, shortlisting, booking, review, and admin-governed approval; and (iv) a premium UI featuring glassmorphism cards, staggered animations, skeleton loading states, and full dark/light theming.

All 20 functional test cases passed successfully in unit, integration, and usability testing — achieving a usability satisfaction score of 4.3/5.0. EVORA demonstrates that a thoughtfully engineered mobile application combining role-based access control, domain-specific AI intelligence, and a rich feature set can substantially improve the efficiency, intelligence, and user experience of event planning — making it a compelling contribution to the intersection of mobile computing and AI-driven service platforms.

Keywords: Flutter, Event Management, Vendor Management, AI Chatbot, Budget Intelligence, MVVM, Provider, Role-Based Access Control, Cross-Platform, Dart, Mobile Application, NLP.

Chapter 1
Introduction

1.1 Purpose of the System

The primary purpose of EVORA is to deliver a comprehensive, intelligent, and role-based mobile environment that revolutionises the traditional event management life-cycle. Contemporary event planning relies heavily on disjointed software applications, manual data entry, and fragmented communication channels. Organizers lack real-time visibility into vendor reliability, budgetary constraints, and task progression. EVORA bridges these gaps by providing an all-encompassing platform catering directly to event organizers, service vendors, and platform administrators through a singular codebase.

By leveraging Google's Flutter framework, EVORA ensures seamless cross-platform performance across both Android and iOS devices. The system is architected to bring computational intelligence into the event planning space through the "Nanban" AI chatbot and a rule-based budget recommendation engine. Ultimately, EVORA aims to demystify complex event coordination, reduce financial overspending, standardise vendor discovery, and elevate the user experience through modern design paradigms and automated assistance.

1.2 Detailed Problem Statement

The contemporary event management space, spanning weddings, corporate summits, and social ceremonies, presents multifaceted challenges that impede efficiency:

  • Fragmented Ecosystems: Organizers are forced to utilise spreadsheets (Excel) for finances, messaging apps (WhatsApp) for vendor communication, and generic task lists (Trello) for deadlines. This lack of integration leads to severe data silos and miscommunication.
  • Absence of Intelligent Financial Guidance: Novice organizers typically lack domain knowledge regarding cost distribution across service categories. Existing platforms fail to provide proactive, data-driven budget allocations, leading to gross over-estimations or mid-event budget deficits.
  • Inefficient Vendor Discovery: Procuring vendors (e.g., caterers, decorators, venues) demands exhaustive web searches, unverifiable reviews, and repetitive price negotiations. Organizers lack a trusted, unified marketplace containing vetted professionals.
  • Lack of Specialized AI Assistance: While generic conversational AI models exist, there are no domain-specific intelligent agents integrated directly within event management software capable of offering instantaneous, context-aware planning advice.
  • Absence of Governance Workflows: Platform-driven markets often suffer from low-quality vendor listings due to an absence of strict administrative approval pipelines.

1.3 Motivation

The commercial events industry represents a multi-trillion-dollar global market. In India, the wedding and corporate event sectors constitute a massive economic engine, yet digital penetration remains remarkably low among independent service providers. This digital disparity motivates the creation of EVORA—a platform designed to empower millions of unorganised sector vendors by granting them digital visibility and booking management capabilities via mobile computing.

Simultaneously, the widespread proliferation of smartphones and high-speed internet provides a fertile foundation for a mobile-first SaaS intervention. The success of AI-driven recommendation logic in e-commerce and streaming media underscores consumer readiness for intelligent algorithms. By combining artificial intelligence with event management, EVORA is positioned to automate the tedious aspects of coordination, allowing users to focus on creative execution.

1.4 Scope of the Project

The EVORA platform constitutes a complete ecosystem with the following functional boundaries:

  • Role-Based Access Control (RBAC): Secure, differentiated interfaces for Administrators, Organizers, and Vendors.
  • Event Operations: Multi-event creation, milestone tracking, and linked checklists.
  • Vendor Marketplace: Discovery and filtering across 13 diverse service categories, comprehensive vendor profiles, dynamic pricing, and a shortlisted/booking transaction model.
  • Budget Intelligence: Smart financial tracking with event-specific percentage allocations, visual expense composition graphs, and over-budget algorithmic warnings.
  • Nanban AI Agent: An onboard chatbot implementing NLP intent-matching to guide users on vendor selection and budget distribution.
  • Integrated Telemetry: In-app threaded messaging between organizers and vendors, securing a chronological record of negotiations.
  • Admin Governance: A dedicated console for holistic platform analytics, user registry management, and mandatory vendor verification workflows.

1.5 Objectives of the Project

The key objectives formulated to address the problem statement include:

  • Cross-Platform Unification: To develop a high-performance Flutter mobile application that compiles natively to iOS and Android, ensuring maximal market reach.
  • Robust State Architecture: To construct a highly scalable Provider-based MVVM (Model-View-ViewModel) architecture ensuring strict separation of business logic from UI rendering.
  • AI Integration: To engineer the "Nanban" chatbot, equipping it with keyword-based intent detection algorithms to interactively resolve user queries.
  • Financial Automation: To implement a budget recommendation engine that structurally assigns predictive funding caps to distinct event service categories based on event topology.
  • Marketplace Fluidity: To establish a seamless vendor booking pipeline encompassing discovery, shortlisting, transaction initiation, and rating evaluation.
  • Administrative Hegemony: To enforce quality control using a rigid administrative approval matrix, ensuring only vetted vendors populate the marketplace grid.

Chapter 3
System Analysis

3.1 Existing Systems and Limitations

An extensive audit of current commercial software utilized within the event management vertical reveals a highly fragmented digital landscape. Organizers typically employ a combination of disparate platforms, leading to severe cognitive overhead and operational latency.

1. Pure Ticketing Platforms (Eventbrite, Meetup):
These B2C applications excel merely at consumer event marketing and ticket allocation. They offer zero backend operational tooling for calculating complex categorical budgets, vendor negotiation, or localized task management. Consequently, they function entirely outside the logistical purview of the organizer.

2. Directory Marketplaces (WedMeGood, WeddingWire):
These robust directories supply extensive portfolios of vendors specific to particular niches (weddings). However, they serve predominantly as glorified search engines. Crucial functions—such as integrated financial modeling, dynamic AI-driven advice, and holistic post-event checklist linkages—are entirely absent. They fail to operate as comprehensive project trackers.

3. Broad Project Managers (Trello, Asana, Excel):
Generic task boards and spreadsheet formulae are endlessly adaptable but infinitely rigid. Organizers must manually construct spreadsheets to monitor expenditures against total capital arrays. Spreadsheets provide zero intelligence, cannot natively interface with vendor catalogues, lack threaded communication features, and are infamously hostile within mobile display bounds.

3.3 Proposed System (EVORA)

EVORA is mathematically and architecturally proposed to eradicate these limitations by serving as a monolithic, intelligent, tri-role conduit encompassing the complete lifecycle of vendor procurement and event realization. Rather than operating as another disjointed directory, EVORA merges the rigidity of corporate spreadsheet tracking with the accessibility of modern B2C marketplaces.

The platform inherently segregates interface logic based on the interacting user. Administrators govern the platform via comprehensive analytics boards and vendor authentication queues. Vendors manipulate commercial portfolios, examine incoming bookings, and traverse monthly service calendars. Organizers seamlessly initiate events, query the Nanban AI agent regarding fiscal allocations, assign vendor-linked checklist tasks, and transmit direct messages to prospective partners.

3.4 Advantages of the Proposed System

  • Total Unification: Consolidates budget metrics, vendor interaction, task tracking, and administrative governance within a singular digital namespace.
  • Predictive Budget Mastery: Employs algorithmic, event-type-specific heuristics to generate intelligent warnings, drastically curbing expenditure bloat.
  • Streamlined Sourcing via AI: Integrates zero-latency conversational bot paradigms providing heuristic-driven insights that automate introductory planning phases.
  • Governed Marketplace Trust: A robust administrative bottleneck ensures qualitative supremacy and eliminates fraudulent vendor permeation via an Approve/Reject triage workflow.
  • Architectural Scalability: Conceived utilizing the robust, decoupled MVVM architecture governed by Flutter's Provider framework, enabling rapid migration toward robust NoSQL/SQL cloud integration layers.
  • Cross-OS Omnipresence: The single-language Dart codebase drastically compresses overhead, engineering a uniform layout synchronously compiled for both iOS frameworks and Android partitions.

The structured database schema driving EVORA enforces referential integrity across the platform's multi-stakeholder ecosystem. The core table, USERS, centrally manages all authentication logic and maps individuals to specific platform authorization roles (Admin, Organizer, Vendor).

Rather than duplicating vendor entities, the VENDORS table establishes a one-to-one foreign key relationship mapping back to the USERS table for authentication, whilst storing professional commercial metrics (categories, dynamic pricing vectors, normalized 0.0–5.0 ratings, and Admin-controlled statuses: pending, approved, or rejected).

The EVENTS table serves as the primary pivot. Each event dictates a one-to-many relationship with the EXPENSES and TASKS tables, assuring that budgets and checklists remain contextually isolated. Consequently, the BOOKINGS junction table successfully links isolated events with discrete vendors, completing the B2B transaction cycle while maintaining strict normalization and querying efficiency.

EVORA is constructed upon a highly decoupled, three-tier Provider-based MVVM (Model-View-ViewModel) paradigm. This architecture explicitly isolates the graphical user interface from the underlying business logic and data persistence layers. The top tier, the Presentation Layer, consists entirely of stateless and stateful Flutter widgets defining the Admin, Organizer, and Vendor modules. These widgets do not contain calculation algorithms or navigation state; they merely observe state objects.

The Business Logic Layer acts as the ViewModel, composed of seven specialized `ChangeNotifier` classes (Providers) securely wrapped around the root `MaterialApp` via a `MultiProvider`. These providers handle complex computations—such as the AI intent parsing and budget percentage distribution—and subsequently broadcast `notifyListeners()` to trigger targeted, localized UI rebuilds without necessitating full-screen re-rendering.

Finally, the Data Layer enforces strict typing through native Dart Models (e.g., `EventModel`, `VendorModel`) and currently interacts with an asynchronous mock storage mechanism. This strictly layered approach ensures that integrating a live NoSQL database like Firebase in the future will require zero modifications to the UI layer, confirming the system's long-term enterprise scalability.

The Use Case diagram delineates the explicit functional boundaries enforced by EVORA's Role-Based Access Control (RBAC) architecture. By categorizing system capabilities against three primary actors—Admin, Organizer, and Vendor—the platform ensures strict operational isolation. A user authenticated as an Organizer cannot query the administrative approval queues, thereby preserving marketplace integrity.

The Organizer operates as the system's primary kinetic actor, possessing access to the widest array of computational tools. Their use cases are interconnected: an Organizer queries the AI Chatbot to understand budget distributions, uses those limits to create a Budget Breakdown, subsequently browses the Vendor Marketplace, and finalizes Bookings. This interdependent functional loop forms the core event-planning workflow of the application.

Conversely, the Admin and Vendor assume reactive governance and service-provisioning roles, respectively. Administrators act as the security bottleneck, utilizing their distinct use cases to sanitize the vendor pool preventing unverified entities from exposing services to the Organizers. Vendors interact sequentially to adjust portfolio aesthetics, respond to inbound organizer inquiries via the messaging module, and confirm calendar bookings.

Chapter 7
System Implementation

7.1 Authentication & Role Management

EVORA implements a robust, state-driven Authentication module structured around Role-Based Access Control (RBAC). The entry sequence begins at the `RoleSelectionScreen`, which injects the selected enumerated role (Admin, Organizer, or Vendor) into the global `UserProvider`. This architecture enforces zero cross-pollination among roles. When a login is executed via `LoginScreen`, the navigation hierarchy immediately segregates routing logic: an authenticated Admin is hard-routed to `AdminDashboard`, while an Organizer is directed toward the `OrganizerDashboard.dart` router array.

Authentication forms implement rigorous, localized input validation using Flutter's native `FormState` mechanics combined with asynchronous error handling. The login procedure simulates backend network payload latencies by utilizing `Future.delayed(Duration(seconds: 1))` whilst projecting an asynchronous `CircularProgressIndicator`, ensuring a standard commercial User Experience (UX).

7.2 Organizer Module

The Organizer namespace constitutes the most computationally dense partition of EVORA. It centers around a multifaceted `OrganizerDashboard` encompassing several critical sub-systems:

  • Event Creation & Management (`CreateEvent`): Organizers input granular event meta-data, critically dictating the event topology (Wedding, Birthday, Corporate). This typography parameter dynamically seeds the Budgeting Engine. All events are stacked temporally within a `ListView.builder` on the dashboard.
  • Budget Overview (`BudgetOverview.dart`): Upon event expansion, financial logic governs the display of categorical distributions. Users input real expenditures (e.g., ₹50,000 for Catering), scaling against algorithmically derived predictive caps, mathematically calculating percentage variances. Over-budget deviations immediately shift corresponding bar chart vectors from positive-green to warning-red.
  • Vendor Discovery (`VendorList` & `VendorDetails`): The system exposes 13 diverse vendor categories. Activating a category fetches a filtered stream from `VendorProvider`, ignoring any vendor not tagged `VendorStatus.approved`. The details view incorporates pricing metrics, an actionable 'Shortlist' toggle, and a finalizing 'Book Vendor' transaction pipeline.
  • Task Management (`EventChecklist`): A contextualised to-do repository natively tied to the parent Event UUID. Tasks incorporate Boolean completion toggles mapped iteratively toward an overarching progression-bar visualisation tracking milestone completion percentages.

7.3 Vendor Module

The Vendor ecosystem focuses entirely on portfolio marketing and reservation fulfillment. Designed specifically for localized, unorganized service providers, the UI rejects complexity in favor of overt simplicity.

  • Portfolio Management (`VendorProfile`): Provides the capacity for vendors to dictate their base pricing tier, service category denomination, and geographic operational limits. It also dynamically visualises an average aggregate rating normalized from 0.0 to 5.0.
  • Actionable Booking Pipeline (`VendorBookings`): When an Organizer triggers a booking, the transaction surfaces instantaneously upon the Vendor board as `PENDING`. Vendors wield the agency to accept or decline algorithmic requests depending entirely upon their availability calendars, ensuring zero occurrences of double-booking overages.
  • Messaging Construct (`OrganizerMessages`): Organizers and Vendors are bridged through a direct, threaded dialogue box isolated uniquely for the negotiation of specific booking instances, thereby nullifying the prerequisite for external applications like WhatsApp.

7.4 Admin Module

The administrative console, localized in `AdminDashboard.dart`, abstracts high-level governance. The architecture centers upon analytical oversight and platform sanitization. Global statistics regarding comprehensive user aggregation, generated revenue thresholds, and absolute booking permutations are aggregated via loop iterations traversing the Providers.

Crucially, the Admin oversees the Pending Vendor Queue (`AdminVendors.dart`). New registrations are placed into holding arrays pending a manual review pipeline. The Admin scrutinizes incoming pricing structures, service validations, and portfolio verifications, triggering either `updateVendorStatus(approved)` or `updateVendorStatus(rejected)`. Rejected arrays are instantly unmounted from global visibility trees, structurally enforcing an enterprise-grade standard of marketplace trust and quality.

7.5 State Management Architecture

To avert the notorious 'Widget Tree Prop-Drilling' anti-pattern, EVORA relies extensively upon `Provider` (`ChangeNotifier`). Seven distinct ChangeNotifiers orchestrate asynchronous global system states (`UserProvider`, `EventProvider`, `VendorProvider`, `BudgetProvider`, `TaskProvider`, `AdminProvider`, and `AppThemeProvider`).

Rather than invoking full-screen `setState()` redraws—which critically degrade framerate performance beneath 60FPS—localized elements employ tightly scoped `Consumer` widgets. Ergo, when the Theme Provider executes `toggleTheme()`, solely those layout widgets reliant on primary colormetrics selectively recalculate against `AppTheme.darkTheme` dictionaries, preserving immense operational bandwidth.

Chapter 8
AI Components & Intelligence

8.1 Nanban Chatbot Logic (Intent Detection)

Nanban, translating colloquially to 'Friend', is EVORA's integrated domain-specific conversational AI. Rather than relying on cloud-bound, token-expensive Large Language Models which risk network timeouts in lower-connectivity Indian geographies, Nanban utilizes an on-device, rule-based Natural Language Processing Intent Engine.

When the Organizer inputs a freeform string query, the engine standardizes the casing (`.toLowerCase()`) and sanitizes punctuation delimiters. The resulting token array is pushed through a cascading switch-statement matrix utilizing subset pattern matching. Nanban successfully resolves queries across four primary classifications:

  • Fiscal Routing ("budget", "cost", "money", "expensive"): Diverts conversational flow to algorithmic budget template generators.
  • Vendor Querying ("suggest", "find", "venue", "caterer", "photographer"): Instigates internal iterations across the `VendorProvider`, returning the highest-rated approved vendor structurally matching the desired metric.
  • Greeting/Politeness ("hey", "hi", "nanban", "thanks"): Supplies conversational anchoring and operational readiness statements.
  • Fallback Catch-All: Returns structured disambiguation prompts (e.g., "Machi, I'm tuned specifically for events...") if the string completely fails the intent match validation matrix.

8.3 Vendor Recommendation Logic

The marketplace utilizes an embedded heuristic sorting algorithm designed to minimize visual clutter. Organizers querying standard lists or requesting direct Chatbot injections do not receive a randomized list of vendors. The VendorProvider applies a multi-pass sorting function prior to UI dissemination. Initially, developers implemented an iteration logic separating `pending` and `rejected` entities out of scope. The remaining `approved` matrix is subsequently ordered descendingly primarily by chronological positive ratings (0 to 5.0) and secondarily bounded by the underlying event's remaining unspent nominal budget.

8.4 Machine Learning Extensions

While the existing zero-latency procedural logic fulfills all baseline operational requirements, the architectural separation guarantees seamless extensibility into advanced Machine Learning strata. Future enhancements dictate mapping the `bookVendor()` functional node to a cloud storage tensor. By capturing thousands of combinatorial booking variables across the userbase, algorithms utilizing Collaborative Filtering (CF) arrays can eventually be layered atop the Vendor recommendation pipeline. This implementation will actively study an organizer's historical propensity to hire premium-class photographers when budgeting high-tier catering, autonomously predicting and structuring recommendations long before the user initiates a manual search.

Chapter 9
Software Description

9.1 Flutter Framework & Dart

Flutter is Google's open-source UI software development kit dedicated to compiling natively rendered, multi-platform applications directly from a single Dart codebase. Unlike React Native which relies on JavaScript OEM bridges, Flutter's architecture revolves around a widget composition model rendered intrinsically by its Skia/Impeller graphics engine. EVORA utilises Flutter precisely to circumvent platform-specific UI fragments, guaranteeing a pixel-perfect, 60fps synchronous experience across heterogeneous Android and iOS displays.

9.2 Material 3 & Glassmorphism Aesthetics

EVORA implements Google's Material 3 design lexicon heavily customised via the AppTheme provider. Instead of flat, utilitarian blocks, EVORA layers a Glassmorphism UI Paradigm over animated backdrops. By compounding the BackdropFilter (ImageFilter.blur) widget with semi-transparent, alpha-channel reduced `#FFFFFF` container containers, the application simulates frosted glass depth. This visual vocabulary dramatically improves cognitive perception, mimicking premium enterprise SaaS optics while preserving absolute legibility.

9.3 Animation Orchestration

To eliminate static transitions and foster fluid interactivity, EVORA integrates a multi-tier animation architecture:

  • Staggered Entry (flutter_staggered_animations): Dynamic datalists (Vendor Portfolios, Upcoming Events) are wrapped inside an AnimationLimiter. Consequently, list items materialize mathematically cascading via a 375ms sequenced `FadeIn` and `SlideAnimation` effect, establishing contextual hierarchy rather than jarring instantaneous loads.
  • Skeletal Shimmer (shimmer): Rather than presenting abstract spinning indicators during `FutureBuilder` loading delays, EVORA implements placeholder geometry replicating structural layouts. An 800ms gradient shimmer effect sweeps continuously across these bounds until genuine JSON parsing fulfills the UI constraint map.
  • Implicit Gestures: Custom `AnimatedPressable` widgets encapsulate standard buttons. Interaction interpolates a sub-second scale transformation (0.95 ratio) simulating visceral, physical depression heuristics.

EVORA mandates a multi-dimensional software validation strategy focusing distinctly upon functional correctness, state isolation, data rendering constraints, and conversational intent accuracy. Executed natively utilizing Flutter's `flutter_test` library. Utilizing the `WidgetTester` dependency, core architectural composites (such as the `GlassCard`, `EVORAHeader`, and `SkeletonLoader`) were instanced directly into headless virtual environments. Automated traversal scripts simulated genuine user operability. An emulated Organizer initiates application load, performs Role Selection, creates an Event tree, inputs budgetary variables, fetches a Vendor matrix, queries "Nanban" regarding financial strategy, and completes a Vendor booking sequentially. This confirmed state preservation horizontally traversing separate module boundaries.

4. Usability Testing

A controlled deployment group encompassing 10 peer students assumed differing platform roles. Success heuristics were monitored against specific task completion latencies (e.g., "Find a Photographer under ₹50,000"). Chapter 11
Conclusion & Future Enhancements

11.1 Conclusion

The research, design, and computational realization of EVORA (Intelligent Event & Vendor Management System) effectively substantiates the hypothesis that integrating conversational AI and predictive financial heuristics into a monolithic, role-segregated mobile architecture drastically optimizes the event management lifecycle.

By shifting away from legacy web-centric directories toward a reactive Flutter MVVM architecture, the platform delivers an uncompromised 60FPS UI spanning heavily animated glassmorphism interfaces across both Android and iOS ecosystems. The rigid instantiation of Role-Based Access Control (RBAC) successfully isolates the Administrator's imperative governance metrics, the Organizers expansive planning suite, and the Vendors commercial portfolio, completely mitigating traditional marketplace latency and fraudulent permeation.

Operationally, EVORA transcends basic directory functionalities by embedding structural intelligence. The Intelligent Budget Provider curtails chronic financial overspending by deploying algorithmic percentage allocations bound algorithmically to specific event typologies. Furthermore, the localized Nanban AI Agent demonstrates the viability of executing robust, on-device NLP intent-parsing to deliver immediate conversational guidance regarding vendor curation.

Ultimately, EVORA serves as a definitive architectural blueprint for modernizing Indias colossal yet deeply unorganized micro-event vendor landscape, converting analog negotiations into a secure, trackable, and intelligently guided digital ecosystem.

11.2 Future Enhancements

While EVORA fulfills its baseline objectives successfully, its scalable architecture accommodates immediate enterprise-grade expansions:

1. Cloud Storage & Backend Infrastructure (Firebase/Supabase)

Replacing the current synchronous mock payload arrays with a real-time NoSQL Firestore database or PostgreSQL Supabase backend. This guarantees dynamic multi-device state synchronization, persistent authentication protocols, and cloud-hosted historical analytics indispensable for long-tail user retention.

2. Machine Learning Collaborative Diagnostics

Upgrading the deterministic vendor heuristic engine with a cloud-trained Collaborative Filtering (CF) ML tensor. The system would actively learn from historical booking combinations (e.g., specific photographers frequently booked alongside premium venues) to supply autonomously predictive suggestions scaling beyond linear constraint models.

3. Direct Unified Payment Gateway Transactability

Integrating robust financial SDKs (Razorpay or Stripe) directly into the `bookVendor()` execution flow. This establishes secure escrow functionality, partial advance-payment capturing logic, and automated digital ledger invoicing—transitioning the app from a discovery conduit into a definitive financial transactor.

4. Push Notification Telemetry

Utilizing Firebase Cloud Messaging (FCM) to trigger background OS-level alert nodes. This ensures immediate user engagement whenever a vendor mutates a booking `status` enum, or when the Budget Engine calculates an impending financial overrun independently of application foreground status.

5. Generative LLM Integration (Gemini Flash)

Evolving the 'Nanban' intent-engine from a localized regex-bound procedural parser into an API-linked conversational avatar powered by Google Gemini. This expansion allows the AI to holistically ingest the user's entire JSON-formatted event state and provide infinitely fluid, unstructured multi-paragraph operational advice.

6. Geospatial Filtering Vectors

Implementing Google Maps API interpolation. Vendors would be strictly bounded by longitude/latitude arrays. The search algorithm would execute Haversine dimensional calculations, prioritizing or exclusively restricting vendor discovery solely to the organizer's proximate geographic perimeter.

7. Multilingual Subsystem Injections

Abstracting all direct hardcoded UI strings into translation maps via the `intl` package, dynamically translating the interface based on the device's native OS locale state. Supporting Hindi, Tamil, Telugu, and Kannada exponentially scales user adoption within Indias tier-2 logistical markets.
